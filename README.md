# GlyphGen

[中文版 README](README.zh-CN.md)

Suggested repository name: `glyphcomposer`

GlyphGen is a structured glyph generation research prototype for graduation projects.
It turns a pair of components plus a binary layout structure into a grayscale character image.

The repository is designed around four workflows:

1. Build a supervised dataset from fonts and decomposition tables.
2. Train a baseline conditional U-Net generator.
3. Train a VAE and latent diffusion model with the same condition interface.
4. Run evaluation, command-line inference, and a Gradio demo.

## Task Definition

The project solves a constrained glyph synthesis problem instead of open-domain text-to-image.

- Input: `component_a`, `component_b`, `structure`, optional `style_id`
- Output: one `256x256` grayscale glyph image
- Supported input modes:
  - `text_component`: components are Unicode strings rendered from a font
  - `image_component`: components are grayscale glyph crops, used for OOV adaptation

The default structure label set is:

- `left_right`
- `top_bottom`
- `full_surround`
- `surround_from_above`
- `surround_from_below`
- `surround_from_left`
- `surround_from_upper_left`
- `surround_from_upper_right`
- `surround_from_lower_left`

## Quick Start

1. Create a virtual environment.
2. Install dependencies from `requirements.txt`.
3. Prepare a font directory and a decomposition CSV.
4. Build the dataset manifest and images.
5. Train the baseline model.
6. Optionally train the VAE, diffusion model, structure probe, and OOV LoRA adapter.

Example commands:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

python scripts/build_dataset.py \
  --decomposition-csv assets/decompositions_sample.csv \
  --font-path /System/Library/Fonts/Hiragino\ Sans\ GB.ttc \
  --output-dir data/generated/sample_dataset

python scripts/train_baseline.py \
  --config configs/baseline.yaml \
  --manifest data/generated/sample_dataset/manifests/train_random.jsonl \
  --val-manifest data/generated/sample_dataset/manifests/val_random.jsonl

python scripts/infer.py \
  --checkpoint checkpoints/baseline/baseline_last.pt \
  --component-a 犭 \
  --component-b 句 \
  --structure left_right \
  --output outputs/dog.png
```

Main training sequence:

```bash
python scripts/train_vae.py \
  --config configs/vae.yaml \
  --manifest data/generated/sample_dataset/manifests/train_random.jsonl \
  --val-manifest data/generated/sample_dataset/manifests/val_random.jsonl

python scripts/train_diffusion.py \
  --config configs/diffusion.yaml \
  --manifest data/generated/sample_dataset/manifests/train_random.jsonl \
  --val-manifest data/generated/sample_dataset/manifests/val_random.jsonl \
  --vae-checkpoint checkpoints/vae/vae_best.pt

python scripts/train_structure_probe.py \
  --config configs/probe.yaml \
  --manifest data/generated/sample_dataset/manifests/train_random.jsonl \
  --val-manifest data/generated/sample_dataset/manifests/val_random.jsonl
```

## Data Format

Each manifest row stores:

- `target_char`
- `font_id`
- `glyph_image`
- `structure`
- `component_a`
- `component_b`
- `component_a_image`
- `component_b_image`
- `style_id`
- `split`

V1 only keeps characters whose decomposition table contains exactly two leaf components and one binary structure tag.

For non-Unicode or OOV characters, use image-conditioned preparation:

```bash
python scripts/prepare_oov_dataset.py \
  --csv assets/oov_image_samples_template.csv \
  --output-dir data/generated/oov_dataset
```

## Notes

- The repository does not ship font files.
- The sample decomposition CSV is intentionally tiny and exists only to validate the pipeline.
- The OOV template expects you to provide target glyph images and component images manually.
- The final glyph image must always be generated by a model. Rules are used only for rendering supervision data and layout heatmaps.
